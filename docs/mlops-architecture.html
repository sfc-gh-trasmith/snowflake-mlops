MLOps Workflow Architecture

End-to-End Fraud Detection Pipeline on Snowflake

Overview

This project implements a production-grade MLOps pipeline for real-time fraud detection on Snowflake. Given a customer's profile and transaction history, the system predicts the probability of a transaction being fraudulent.

Use Case

Binary classification: Is this transaction fraudulent?

Model: XGBoost classifier
Features: 12 customer-level aggregations
Data: 100K transactions, 3% fraud rate

Environments

DEV Local experimentation

STAGE CI validation (automated)

PROD Production serving

Key Technologies

Snowflake Feature Store (Dynamic Tables)
ML Jobs (@remote on Compute Pools)
Model Registry + Replication
SPCS Inference with REST Gateway
GitHub Actions with OIDC

Key Design Principle: Source data always lives in PROD. DEV and STAGE read from PROD for training but write artifacts (features, models, experiments) to their own databases. PROD never trains -- it only receives model replicas from STAGE.

ML Pipeline Steps

1. Feature Store (Dynamic Tables)

Customer-level risk features computed as Snowflake Dynamic Tables with hourly auto-refresh:

ANNUAL_INCOMEReported annual incomeProfile

2. Model Training (ML Job on Compute Pool)

Algorithm: XGBoost with scale_pos_weight=33 (handles 3% fraud class imbalance)

Validation: 5-fold stratified cross-validation

Compute: CPU_X64_M instance family, 1-3 nodes auto-scaling

Execution: @remote decorator submits to Snowflake Compute Pool

Metrics: AUC-ROC, PR-AUC, Precision, Recall, F1

3. Model Registry

Models registered with version tracking, dependency declaration, input schema inference, and deployment metadata.

4. Model Replication (STAGE to PROD)

CREATE MODEL SNOW_MLOPS_PROD.ML.MLOPS_FRAUD_DETECTOR
  WITH VERSION V1
  FROM MODEL SNOW_MLOPS_STAGE.ML.MLOPS_FRAUD_DETECTOR
  VERSION V1;

Intra-account, cross-database copy. Atomic operation preserving all metadata and dependencies.

5. Inference Service (SPCS + REST Gateway)

Ingress: Public REST endpoint (ingress_enabled=True)

Scaling: Up to 2 instances horizontal

Methods: predict (binary) and predict_proba (probability)

Auto-suspend: 30 minutes idle timeout

Key Design Decisions

  1. Environment separation by database -- Same account, different databases. Simple RBAC.
  2. Source data always in PROD -- Single source of truth for raw data.
  3. No service in STAGE -- Validates training only; service tested in DEV and deployed in PROD.
  4. Model replication via SQL -- Atomic cross-database copy, no pickle files shipped.
  5. OIDC for CI/CD -- Zero stored secrets, short-lived tokens, environment-scoped.
  6. Manual PROD gate -- workflow_dispatch prevents accidental deployments.
  7. @remote for training -- Runs at scale inside Snowflake, not on CI runners.
