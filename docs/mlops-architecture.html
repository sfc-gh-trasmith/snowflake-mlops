1. What is MLOps on Snowflake?

MLOps is the practice of operationalizing machine learning -- making models repeatable, auditable, and automatically deployable. Snowflake provides native services for every step of the ML lifecycle, eliminating the need for external tools like MLflow, SageMaker, or Kubernetes.

The Snowflake ML platform is not a single product -- it's a set of integrated capabilities that work together: Feature Store (Dynamic Tables), ML Jobs (Compute Pools), Model Registry, SPCS (container services), and Gateways. Each solves one part of the problem. This guide shows how they compose into a complete MLOps pipeline.

The Use Case: Fraud Detection

We'll trace through a real implementation that scores customer transactions for fraud probability in real-time. The system:

  • Computes 12 customer-level risk features from transaction history
  • Trains an XGBoost classifier handling 3% class imbalance
  • Serves predictions via REST API with sub-second latency
  • Auto-deploys new model versions with zero downtime

Everything stays inside Snowflake. Data never leaves the platform -- features are computed in-place via SQL, training reads directly from feature tables, and the model is served from Snowflake containers. This eliminates data movement, simplifies security, and gives you a single governance boundary.

2. Feature Store (Dynamic Tables)

What It Is

Snowflake's Feature Store is built on Dynamic Tables -- SQL-defined transformations that Snowflake keeps fresh automatically. You declare the transformation once; Snowflake handles scheduling, incremental refresh, and serving.

A Feature View is not a batch job you schedule -- it's a declaration of what features should look like. Snowflake observes the upstream data (via change tracking) and refreshes the Dynamic Table when data changes, up to the target lag you specify. Think of it like a materialized view that's always within N minutes of current.

How It Works in This Project

-- This is a Feature View backed by a Dynamic Table
-- Target lag: 60 minutes (refreshes when source data is >60 min stale)
-- Reads from: RAW_TRANSACTIONS + CUSTOMER_PROFILES
-- Outputs: 12 customer-level risk features

CREATE OR REPLACE DYNAMIC TABLE CUSTOMER_RISK_FEATURES
  TARGET_LAG = '60 minutes'
  WAREHOUSE = SNOW_MLOPS_DEV_WH
AS
SELECT
    t.CUSTOMER_ID,
    COUNT(*)                          AS TOTAL_TXN_COUNT,
    AVG(t.AMOUNT)                     AS AVG_TXN_AMOUNT,
    MAX(t.AMOUNT)                     AS MAX_TXN_AMOUNT,
    STDDEV(t.AMOUNT)                  AS STDDEV_TXN_AMOUNT,
    COUNT(DISTINCT t.MERCHANT_ID)     AS UNIQUE_MERCHANTS,
    SUM(CASE WHEN t.IS_FRAUD=1 THEN 1 ELSE 0 END) AS HISTORICAL_FRAUD_COUNT,
    AVG(t.IS_FRAUD)                   AS HISTORICAL_FRAUD_RATE,
    COUNT(DISTINCT DATE(t.TXN_TIME))  AS ACTIVE_DAYS,
    AVG(CASE WHEN HOUR(t.TXN_TIME) BETWEEN 0 AND 5 THEN 1.0 ELSE 0.0 END)
                                      AS LATE_NIGHT_TXN_RATIO,
    c.CREDIT_SCORE,
    DATEDIFF('day', c.ACCOUNT_OPEN_DATE, CURRENT_DATE()) AS ACCOUNT_AGE_DAYS,
    c.ANNUAL_INCOME
FROM RAW_TRANSACTIONS t
JOIN CUSTOMER_PROFILES c ON t.CUSTOMER_ID = c.CUSTOMER_ID
GROUP BY t.CUSTOMER_ID, c.CREDIT_SCORE, c.ACCOUNT_OPEN_DATE, c.ANNUAL_INCOME;

Feature Design Decisions

FeatureSignal TypeWhy It Matters for Fraud
TOTAL_TXN_COUNTVelocityFraudsters often make many small test charges
AVG/MAX/STDDEV_TXN_AMOUNTSpendingUnusual amounts deviate from normal patterns
UNIQUE_MERCHANTSBehaviorSpread across many merchants = card testing
HISTORICAL_FRAUD_COUNT/RATERiskPast fraud is the strongest predictor of future fraud
LATE_NIGHT_TXN_RATIOTemporalFraud disproportionately occurs at unusual hours
CREDIT_SCOREProfileLower scores correlate with higher fraud risk
ACCOUNT_AGE_DAYSProfileNew accounts are higher risk (less history)
Feature-serving skew. A classic ML failure mode is computing features differently at training time vs serving time. Because we use the same Dynamic Table for both training reads and (potentially) online serving, the computation is guaranteed identical. This is a major advantage of Snowflake's approach over custom feature pipelines.
Why Dynamic Tables instead of a scheduled dbt job? Dynamic Tables are declarative -- you don't manage scheduling, retries, or orchestration. Snowflake tracks upstream dependencies and refreshes automatically. They also support incremental refresh (only processing changed data), which scales better than full table rebuilds.

3. Model Training (ML Jobs)

What It Is

Snowflake ML Jobs let you run arbitrary Python (training, inference, batch processing) on dedicated Compute Pools -- containerized GPU/CPU instances managed by Snowflake. The @remote decorator makes it feel like calling a local function, but it executes inside Snowflake.

The @remote decorator does something unusual: it serializes your function, ships it to a Snowflake-managed container on the compute pool, installs dependencies, executes the function with full Snowpark access, and returns the result. The function runs inside Snowflake's network boundary -- it can access tables directly without data egress. The caller (your laptop or CI runner) just gets back the return value.

How Training Works

from snowflake.ml.jobs import remote

@remote(
    "SNOW_MLOPS_DEV_POOL",                    # Which compute pool to run on
    stage_name="@SNOW_MLOPS_DEV.ML.JOB_STAGE", # Where to store job artifacts
    pip_requirements=[                         # Installed in the container
        "xgboost",
        "scikit-learn",
        "snowflake-ml-python",
    ],
)
def train_and_register() -> str:
    """Runs INSIDE the compute pool container."""

    # Get a session (already authenticated inside Snowflake)
    session = Session.builder.getOrCreate()

    # Read training data directly via SQL (no data movement!)
    df = session.sql("""
        SELECT features.*, labels.IS_FRAUD
        FROM CUSTOMER_RISK_FEATURES features
        JOIN RAW_TRANSACTIONS labels ON ...
    """).to_pandas()

    # Standard scikit-learn/xgboost workflow
    X_train, X_test, y_train, y_test = train_test_split(...)
    model = xgb.XGBClassifier(scale_pos_weight=33, ...)
    model.fit(X_train, y_train)

    # Register to Snowflake Model Registry
    reg = Registry(session=session)
    reg.log_model(model=model, model_name="MLOPS_FRAUD_DETECTOR", ...)

    return json.dumps({"status": "success", "auc": 0.96})

Key Parameters Explained

ParameterPurpose
COMPUTE_POOLThe pool of machines to run on (e.g., CPU_X64_M = 16 vCPU, 64GB RAM)
stage_nameInternal stage where Snowflake stores the serialized function + deps
pip_requirementsPython packages installed in the container (NOT packages=)
Do not use packages= -- this parameter doesn't work with @remote and silently does nothing. Always use pip_requirements=[]. This is a common source of "module not found" errors.
The function takes no arguments. You cannot pass session=session to a @remote function. Inside the container, call Session.builder.getOrCreate() to get an authenticated session. The container inherits the caller's role and context.

Why scale_pos_weight=33?

With ~3% fraud rate, the dataset is heavily imbalanced (97% legitimate, 3% fraud). Without correction, the model would learn to always predict "legitimate" and achieve 97% accuracy while catching zero fraud. scale_pos_weight tells XGBoost to weight fraud samples 33x higher (approximately 97/3), forcing the model to learn fraud patterns.

Why not train on the CI runner? CI runners (GitHub Actions) are small VMs with limited RAM and no direct access to Snowflake data. By using @remote, training happens on a beefy compute pool node with direct SQL access to terabytes of data -- no data extraction step, no CSV exports, no S3 intermediaries. The CI runner just submits the job and waits.

4. Model Registry

What It Is

The Snowflake Model Registry is a versioned object store for trained models -- think Git for ML artifacts. Each registered model is a first-class Snowflake object with versions, metadata, input/output signatures, and deployment capabilities.

A registered model is a database object, not a file. You refer to it as DATABASE.SCHEMA.MODEL_NAME. It has versions (V1, V2, V3...), a default version, aliases, and SQL-level access control. You can GRANT access, DESCRIBE it, and DROP it like any table. This is fundamentally different from MLflow's file-based approach.

Registration Code

from snowflake.ml.registry import Registry

reg = Registry(session=session, database_name="SNOW_MLOPS_DEV", schema_name="ML")

reg.log_model(
    model=model,                              # The trained XGBoost object
    model_name="MLOPS_FRAUD_DETECTOR",        # Name in the registry
    version_name="V3",                        # Auto-incremented
    conda_dependencies=["xgboost", "scikit-learn"],  # For container rebuild
    sample_input_data=X_test.head(10),        # Infers input schema
    comment="AUC=0.96, F1=0.72 | git:a3f8b2c",
)

Auto-Versioning Pattern

Rather than hardcoding version names, each pipeline run queries existing versions and increments:

# Inside the training function:
versions = session.sql("SHOW VERSIONS IN MODEL MLOPS_FRAUD_DETECTOR").collect()
max_v = max(int(v["name"].replace("V","")) for v in versions)
next_version = f"V{max_v + 1}"  # V1 -> V2 -> V3 -> ...
Why auto-increment instead of git SHA? Version names like "V1", "V2" are human-readable and orderable. The git SHA goes in the comment field for traceability. This makes it easy to say "deploy V3" or "rollback to V2" without looking up hashes.

What Gets Stored

MetadataPurpose
Model artifactSerialized XGBoost model (pickle-compatible)
Input signatureColumn names + types (inferred from sample_input_data)
Output signatureWhat predict and predict_proba return
Conda dependenciesExact packages needed to load the model
CommentMetrics, git SHA, timestamp -- your audit trail
Default versionWhich version is "current" (used by deployment)

Cross-Database Replication

Models can be copied between databases atomically:

-- Copy V3 from STAGE to PROD (same account, different databases)
ALTER MODEL SNOW_MLOPS_PROD.ML.MLOPS_FRAUD_DETECTOR
  ADD VERSION V3
  FROM MODEL SNOW_MLOPS_STAGE.ML.MLOPS_FRAUD_DETECTOR VERSION V3;

-- Make V3 the version PROD will serve
ALTER MODEL SNOW_MLOPS_PROD.ML.MLOPS_FRAUD_DETECTOR
  SET DEFAULT_VERSION = V3;
Model replication is not a file copy. It's an atomic Snowflake operation that preserves all metadata, signatures, and dependencies. The model in PROD is a full first-class object with its own version history. If STAGE has V1-V5 but you only replicate V3 and V5, PROD only has those two versions.

5. Real-Time Serving (SPCS + Gateway)

What It Is

Snowflake can deploy any registered model as a containerized REST API running on Snowpark Container Services (SPCS). You get a public HTTPS endpoint that accepts JSON payloads and returns predictions.

create_service() does three things: (1) builds a Docker container image from your model + dependencies, (2) deploys it to a compute pool with HTTP ingress, and (3) registers the endpoint. You never write a Dockerfile, manage a container registry, or configure a load balancer. Snowflake handles all of this from the model artifact alone.

Deploying a Service

mv = reg.get_model("MLOPS_FRAUD_DETECTOR").version("V3")

mv.create_service(
    service_name="MLOPS_FRAUD_DETECTOR_SERVICE_V3",
    service_compute_pool="SNOW_MLOPS_PROD_POOL",  # Where containers run
    ingress_enabled=True,                          # Public HTTPS endpoint
    max_instances=2,                               # Horizontal auto-scaling
)
# Duration: 10-30 min (image build on first deploy, ~2 min on subsequent)

Calling the Endpoint

# POST https://<endpoint>/predict-proba
# Authorization: Snowflake Token="<PAT_TOKEN>"
{
  "dataframe_split": {
    "index": [0],
    "columns": ["TOTAL_TXN_COUNT", "AVG_TXN_AMOUNT", "MAX_TXN_AMOUNT",
                "STDDEV_TXN_AMOUNT", "UNIQUE_MERCHANTS", "HISTORICAL_FRAUD_COUNT",
                "HISTORICAL_FRAUD_RATE", "ACTIVE_DAYS", "LATE_NIGHT_TXN_RATIO",
                "CREDIT_SCORE", "ACCOUNT_AGE_DAYS", "ANNUAL_INCOME"],
    "data": [[20, 150.50, 500.00, 100.0, 10, 0, 0.0, 30, 0.05, 700, 365, 80000]]
  }
}

# Response:
[{"output_feature_0": 0.9998, "output_feature_1": 0.0002}]
# output_feature_0 = P(legitimate)
# output_feature_1 = P(fraud)
Method names become URL paths with hyphens. Python method predict_proba becomes URL path /predict-proba. Underscores are replaced with hyphens in the endpoint URL. This catches people who try /predict_proba and get a 404.

The Gateway: Stable URLs + Zero-Downtime Deploys

A bare SPCS service has a problem: its URL is tied to the service object. When you drop and recreate a service (e.g., for a new model version), the URL changes. Clients break.

The Snowflake Gateway solves this by providing a permanent URL that routes to underlying services:

flowchart LR
    CLIENT[API Clients] --> GW["Gateway (stable URL)"]
    GW -->|"100%"| V3[SERVICE_V3]
    GW -.->|"0% (old)"| V2[SERVICE_V2]
The Gateway is the production-facing URL. Services are implementation details. Clients always hit the gateway URL (which never changes). Behind it, you can swap services, split traffic (90/10 for canary testing), or fail over to healthy endpoints -- all without changing a single client configuration.

Blue/Green Deployment Pattern

# 1. Deploy new service (coexists with old one)
mv.create_service(service_name="SERVICE_V3", ...)

# 2. Wait for READY + health check against SERVICE_V3 directly

# 3. Shift 100% traffic to new service
ALTER GATEWAY FRAUD_DETECTOR_GATEWAY
FROM SPECIFICATION $$
  spec:
    type: traffic_split
    split_type: custom
    targets:
      - type: endpoint
        value: SNOW_MLOPS_PROD.ML.SERVICE_V3!inference
        weight: 100
$$;

# 4. Drop old service (frees resources)
DROP SERVICE SERVICE_V2;
Why blue/green instead of in-place upgrade? In-place upgrades have a gap: the old service is down while the new one starts. With blue/green, the new service is fully operational and health-checked before any traffic moves. If the health check fails, the old service keeps serving. Zero risk, zero downtime.

6. Multi-Environment Promotion

The Three-Environment Pattern

DEV

Purpose: Experimentation

Who: Developers (local)

Training: Yes

Service: Optional (testing)

Trigger: Manual

STAGE

Purpose: CI validation

Who: GitHub Actions

Training: Yes

Service: Never

Trigger: Auto (merge to main)

PROD

Purpose: Production serving

Who: GitHub Actions

Training: Never

Service: Always

Trigger: Manual + approval

Data Flow Principle

Source data always lives in PROD. DEV and STAGE read from PROD for training but write all artifacts (features, models, experiments) to their own databases. PROD never trains -- it only receives model replicas from STAGE.
Why read from PROD in all environments? Training on different data in each environment is a recipe for "works in dev, fails in prod" scenarios. By always reading from the same source data, you guarantee that the model trained in STAGE will behave identically to what a PROD-trained model would produce.

7. CI/CD Infrastructure

The workflows above rely on two key pieces of infrastructure: OIDC authentication (zero secrets) and network access for CI runners.

OIDC Authentication (Zero Secrets)

No passwords, no API keys, no stored secrets. GitHub Actions authenticates to Snowflake using OIDC (OpenID Connect) -- a protocol where GitHub mints a short-lived JWT token that Snowflake validates directly. The token is generated fresh for each workflow run and expires in minutes. Nothing is stored in GitHub Secrets. 
sequenceDiagram
    participant GH as GitHub Actions
    participant IDP as GitHub OIDC Provider
    participant SF as Snowflake
    GH->>IDP: "I'm running in environment STAGE"
    IDP-->>GH: JWT signed by GitHub
    Note over GH: Token contains subject claim
    GH->>SF: "Here's my JWT, authenticate me"
    SF->>SF: Verify signature + check subject matches user
    SF-->>GH: Session granted (short-lived)
    GH->>SF: Run ML pipeline via session

Environment-Scoped Service Users

Each Snowflake service user is locked to a specific GitHub environment via the OIDC subject claim:

-- STAGE user: can only be authenticated from the STAGE environment
CREATE USER SVC_GITHUB_ACTIONS_STAGE TYPE = SERVICE;
-- Subject: repo:org/repo:environment:STAGE

-- PROD user: can only be authenticated from the PROD environment
CREATE USER SVC_GITHUB_ACTIONS TYPE = SERVICE;
-- Subject: repo:org/repo:environment:PROD
Why OIDC over stored credentials? Stored secrets can leak (commit history, logs, shared access). OIDC tokens are generated at runtime, scoped to a specific workflow run, and expire immediately. Even if somehow intercepted, they can't be replayed. The environment scoping adds a second layer -- a STAGE token literally cannot authenticate as PROD.

Network Access for CI Runners

GitHub Actions runners have dynamic IPs that change every run. Snowflake provides a managed network rule that stays current automatically:

-- Snowflake maintains this rule (IPs updated automatically)
CREATE NETWORK POLICY GITHUB_ACTIONS_POLICY
  ALLOWED_NETWORK_RULE_LIST = ('SNOWFLAKE.NETWORK_SECURITY.GITHUBACTIONS_GLOBAL');

-- Assign to both service users
ALTER USER SVC_GITHUB_ACTIONS SET NETWORK_POLICY = 'GITHUB_ACTIONS_POLICY';
ALTER USER SVC_GITHUB_ACTIONS_STAGE SET NETWORK_POLICY = 'GITHUB_ACTIONS_POLICY';
Without a network policy, CI will fail. If your Snowflake account has a default network policy that restricts IPs, GitHub Actions runners will be blocked. The managed rule GITHUBACTIONS_GLOBAL automatically includes all GitHub runner IPs and is maintained by Snowflake -- you never update it manually.
