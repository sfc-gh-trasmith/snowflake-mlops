MLOps on Snowflake

Framework documentation for building production ML pipelines using Snowflake-native services.

This document describes how to build an end-to-end MLOps pipeline on Snowflake using Feature Store (Dynamic Tables), ML Jobs (Compute Pools), Model Registry, SPCS inference services, and Snowflake Gateway for zero-downtime deployments. The framework is demonstrated with a fraud detection use case but applies to any ML problem.

Feature Store

Features are defined as Snowflake Feature Views backed by Dynamic Tables. You declare the transformation SQL once; Snowflake handles scheduling, refresh, and serving.

How it works

A Feature View is a Dynamic Table that computes features from source data. It refreshes automatically based on a target lag (e.g., 60 minutes). The Feature Store API registers entities, feature views, and versions.

-- Example: customer-level risk features (Dynamic Table)
CREATE OR REPLACE DYNAMIC TABLE CUSTOMER_RISK_FEATURES
  TARGET_LAG = '60 minutes'
  WAREHOUSE = SNOW_MLOPS_DEV_WH
AS
SELECT
    t.CUSTOMER_ID,
    COUNT(*)                          AS TOTAL_TXN_COUNT,
    AVG(t.AMOUNT)                     AS AVG_TXN_AMOUNT,
    MAX(t.AMOUNT)                     AS MAX_TXN_AMOUNT,
    STDDEV(t.AMOUNT)                  AS STDDEV_TXN_AMOUNT,
    COUNT(DISTINCT t.MERCHANT_ID)     AS UNIQUE_MERCHANTS,
    SUM(CASE WHEN t.IS_FRAUD=1 THEN 1 ELSE 0 END) AS HISTORICAL_FRAUD_COUNT,
    AVG(t.IS_FRAUD)                   AS HISTORICAL_FRAUD_RATE,
    COUNT(DISTINCT DATE(t.TXN_TIME))  AS ACTIVE_DAYS,
    c.CREDIT_SCORE,
    c.ANNUAL_INCOME
FROM RAW_TRANSACTIONS t
JOIN CUSTOMER_PROFILES c ON t.CUSTOMER_ID = c.CUSTOMER_ID
GROUP BY ALL;

Feature versioning

Feature View versions are tracked in config.py:

FEATURE_VIEW_NAME = "CUSTOMER_RISK_FEATURES"
FEATURE_VIEW_VERSION = "V1"  # Bump when feature SQL changes

When you bump the version, the next pipeline run creates a new Dynamic Table (CUSTOMER_RISK_FEATURES$V2) and trains against it. The model comment records which feature version it was trained on.

Feature promotion across environments

Feature Views cannot be replicated like models. A Dynamic Table is a live computation, not a static artifact. Promotion means re-running the same registration code against the target database.

EnvironmentFeature ViewRefresh StrategyPurpose
DEVYesOn-demandTraining iteration
STAGEYesOnce per pipeline runCI training validation
PRODNo (offline) / Yes (online)Continuous (if online)Feature lookup at inference

Offline serving (current)

The client sends all features in the inference request. No Feature View needed in PROD.

# CI/CD flow (offline):
# 1. Developer modifies feature SQL → merges to main
# 2. deploy.yml → register_feature_views(STAGE) → creates/updates DT
# 3. Train model on updated features → new model version
# 4. Replicate model to PROD
# 5. Client sends pre-computed features in request

Online serving (future)

The client sends only an entity key. The inference service fetches features from an online Feature Store (Snowflake Postgres) before scoring.

# CI/CD flow (online):
# 1. Developer modifies feature SQL → merges to main
# 2. deploy.yml → register_feature_views(STAGE) → train → replicate
# 3. deploy.yml → register_feature_views(PROD) with OnlineConfig
# 4. deploy-prod.yml → create_service(..., feature_sources_per_function={...})
# 5. Client sends {CUSTOMER_ID: "C123"} → service fetches → scores
Note Online serving requires a Snowflake Postgres instance for low-latency feature lookups. The Dynamic Table syncs to Postgres with ~10s target lag.

Model Training

Training runs on Snowflake Compute Pools using the @remote decorator from snowflake.ml.jobs. The decorator serializes your Python function and executes it inside a managed container.

How it works

from snowflake.ml.jobs import remote

@remote(
    "SNOW_MLOPS_DEV_POOL",
    stage_name="@SNOW_MLOPS_DEV.ML.JOB_STAGE",
    pip_requirements=["xgboost", "scikit-learn", "snowflake-ml-python"],
)
def train_and_register() -> str:
    session = Session.builder.getOrCreate()

    # Read directly from Feature Store Dynamic Tables (no data movement)
    df = session.sql("SELECT ... FROM CUSTOMER_RISK_FEATURES$V1 ...").to_pandas()

    model = xgb.XGBClassifier(scale_pos_weight=33, ...)
    model.fit(X_train, y_train)

    reg = Registry(session=session)
    reg.log_model(model=model, model_name="MLOPS_FRAUD_DETECTOR", ...)

    return json.dumps({"status": "success", "metrics": {...}})

Key parameters

ParameterDescription
COMPUTE_POOLPool of machines to run on (e.g., CPU_X64_M = 16 vCPU, 64GB RAM)
stage_nameInternal stage where the serialized function is stored
pip_requirementsPackages installed in the container at runtime
Important Use pip_requirements=[], not packages=[]. The packages parameter is silently ignored by @remote. The function takes no arguments -- call Session.builder.getOrCreate() inside the container to get an authenticated session.

Job submission

The @remote decorator serializes the function body using cloudpickle, uploads it to the specified stage, and submits it to the compute pool. The caller (your laptop or CI runner) waits for the return value. The function executes inside Snowflake's network boundary with full SQL access.

Model Registry

The Snowflake Model Registry stores trained models as versioned database objects. Each model has versions (V1, V2, ...), a default version, aliases, and SQL-level access control.

Auto-versioning

# Inside the training function:
versions = session.sql("SHOW VERSIONS IN MODEL MLOPS_FRAUD_DETECTOR").collect()
max_v = max(int(v["name"].replace("V","")) for v in versions)
next_version = f"V{max_v + 1}"

Registration

reg.log_model(
    model=model,
    model_name="MLOPS_FRAUD_DETECTOR",
    version_name="V3",
    conda_dependencies=["xgboost", "scikit-learn"],
    sample_input_data=X_test.head(10),
    comment="features:CUSTOMER_RISK_FEATURES$V1 | AUC=0.96 | git:a3f8b2c",
)

The model comment records feature version, metrics, and git SHA for traceability.

Cross-database replication

Models are promoted from STAGE to PROD using SQL replication:

ALTER MODEL SNOW_MLOPS_PROD.ML.MLOPS_FRAUD_DETECTOR
  ADD VERSION V3
  FROM MODEL SNOW_MLOPS_STAGE.ML.MLOPS_FRAUD_DETECTOR VERSION V3;

ALTER MODEL SNOW_MLOPS_PROD.ML.MLOPS_FRAUD_DETECTOR
  SET DEFAULT_VERSION = V3;
Note Model replication is an atomic operation that preserves all metadata, signatures, and dependencies. It is not a file copy.

Real-Time Inference

Registered models are deployed as containerized REST services on Snowpark Container Services (SPCS). Snowflake Gateway provides a stable URL for zero-downtime upgrades.

Deploying a service

mv = reg.get_model("MLOPS_FRAUD_DETECTOR").version("V3")

mv.create_service(
    service_name="MLOPS_FRAUD_DETECTOR_SERVICE_V3",
    service_compute_pool="SNOW_MLOPS_PROD_POOL",
    ingress_enabled=True,
    max_instances=2,
)

This builds a container image (~10 min first time), deploys it to the compute pool, and creates a public HTTPS endpoint.

Calling the endpoint

POST https://<gateway-url>/predict-proba
Authorization: Snowflake Token="<PAT>"
Content-Type: application/json

{
  "dataframe_split": {
    "index": [0],
    "columns": ["TOTAL_TXN_COUNT", "AVG_TXN_AMOUNT", ...],
    "data": [[20, 150.5, 500.0, ...]]
  }
}
Note Method names map to URL paths with hyphens: Python method predict_proba becomes endpoint path /predict-proba.

Snowflake Gateway

The Gateway provides a permanent hostname that routes to underlying services. When services are recreated (model upgrades), the Gateway URL does not change.

flowchart LR
    CLIENT[API Clients] --> GW["Gateway (stable URL)"]
    GW -->|"100%"| V3[SERVICE_V3]

Blue/green deployment

  1. Create new versioned service (SERVICE_V3) alongside the old one
  2. Wait for READY state
  3. Health check the new service directly
  4. ALTER GATEWAY to shift 100% traffic to the new service
  5. Drop the old service
ALTER GATEWAY SNOW_MLOPS_PROD.ML.FRAUD_DETECTOR_GATEWAY
FROM SPECIFICATION $$
  spec:
    type: traffic_split
    split_type: custom
    targets:
      - type: endpoint
        value: SNOW_MLOPS_PROD.ML.MLOPS_FRAUD_DETECTOR_SERVICE_V3!inference
        weight: 100
$$;

Environment Promotion

Code and models flow through three environments with automated validation and approval gates.

Environment configuration

EnvironmentDatabaseTrainingServingTrigger
DEVSNOW_MLOPS_DEVYesOptionalManual
STAGESNOW_MLOPS_STAGEYesNeverAuto (merge to main)
PRODSNOW_MLOPS_PRODNeverAlwaysManual + approval

Source data always lives in PROD. DEV and STAGE read from PROD for training but write artifacts to their own databases.

Promotion flow

flowchart TD
    CODE[Developer writes code] --> PR[Open Pull Request]
    PR --> CHECKS["PR Checks (lint, test)"]
    CHECKS --> MERGE[Merge to main]
    MERGE --> FS[Register Feature Store in STAGE]
    FS --> TRAIN[Train on STAGE_POOL]
    TRAIN --> GATE{Quality Gate}
    GATE -->|pass| REG[Register model]
    REG --> COPY[Replicate to PROD]
    COPY --> DISPATCH["Manual dispatch + approval"]
    DISPATCH --> DEPLOY[Blue/green deploy]
    DEPLOY --> LIVE[Live via Gateway]
    GATE -->|fail| BLOCK[Pipeline blocked]

Quality gate

The STAGE pipeline enforces metric thresholds before promoting to PROD:

MetricThresholdEffect if failed
AUC-ROC>= 0.85Replication blocked
Precision>= 0.70Replication blocked
Recall>= 0.60Replication blocked

Metrics are written to the GitHub Actions Job Summary and posted as a comment on the merged PR.

Task DAG

The ML pipeline is defined as a Snowflake Task (ML_TRAINING_PIPELINE) deployed from git on every merge. This ensures the Task definition always matches the code on main.

# Deploy Task DAG for an environment:
python source/pipeline/ml_pipeline_dag.py --deploy --env stage

# Trigger execution:
python source/pipeline/ml_pipeline_dag.py --run --env stage

Pluggable promotion topology

The model promotion step is implemented as a pluggable strategy controlled by the TOPOLOGY environment variable:

OIDC authentication

GitHub Actions authenticates to Snowflake using OIDC. GitHub mints a short-lived JWT; Snowflake validates the issuer and subject claim. No secrets are stored.

sequenceDiagram
    participant GH as GitHub Actions
    participant SF as Snowflake
    GH->>GH: Mint OIDC JWT
    GH->>SF: Present JWT with subject claim
    SF->>SF: Validate issuer + subject
    SF-->>GH: Session token
    GH->>SF: Execute pipeline

Service users are scoped to specific GitHub environments:

-- STAGE service user
CREATE USER SVC_GITHUB_ACTIONS_STAGE
  TYPE = SERVICE
  DEFAULT_ROLE = MLOPS_DEPLOY_ROLE
  WORKLOAD_IDENTITY = (
    TYPE = OIDC
    ISSUER = 'https://token.actions.githubusercontent.com'
    SUBJECT = 'repo:owner/repo:environment:STAGE'
  );

-- PROD service user
CREATE USER SVC_GITHUB_ACTIONS
  TYPE = SERVICE
  DEFAULT_ROLE = MLOPS_DEPLOY_ROLE
  WORKLOAD_IDENTITY = (
    TYPE = OIDC
    ISSUER = 'https://token.actions.githubusercontent.com'
    SUBJECT = 'repo:owner/repo:environment:PROD'
  );

Network access

GitHub Actions runners use dynamic IPs. Snowflake provides a managed network rule:

CREATE NETWORK POLICY GITHUB_ACTIONS_POLICY
  ALLOWED_NETWORK_RULE_LIST = ('SNOWFLAKE.NETWORK_SECURITY.GITHUBACTIONS_GLOBAL');

ALTER USER SVC_GITHUB_ACTIONS_STAGE SET NETWORK_POLICY = 'GITHUB_ACTIONS_POLICY';
ALTER USER SVC_GITHUB_ACTIONS SET NETWORK_POLICY = 'GITHUB_ACTIONS_POLICY';
